vue打包之后的脚本在IE不兼容(含有ES6语法)

1、IE不兼容报错表现

2、经过排查，发现在引用inc sdk的时候，utils.config部分引用了第三方库cos-nodejs-sdk-v5，这是一个commonjs规范的库，且inc utils模块在使用rollup打包的时候，将cos-nodejs-sdk-v5设置为第三方依赖，并不会在打包的时候，将依赖源码打包进inc sdk中

3、vue项目在采用webpack打包的时候，检测到对@incubator/utils的依赖，同时会将utils.config的依赖关系打包到最后的js文件中。注意，对于采用ES6语法直接发布的依赖包，默认的vue.config配置并不会做ES5转换工作,这也是导致1中IE不兼容ES6语法而报错的原因。比如使用const、箭头函数等语法

4、在vue.config.js中，新增transpileDependencies选项，可以显示指定构建过程中对依赖项进行转换成ES5语法后再构建

5、经过4之后，发现在IE环境下，某个依赖依赖会报错”strict模式下不允许分配到只读属性“,错误指向的是module.export=function(){...}。这个错误原因未知

6、经过对inc utils依赖包的分析，使用cos-nodejs-sdk-v5依赖的utils.config主要用于node端获取远程配置使用，很少在浏览器端使用。所以将此模块拆分出来，放在专门用于node端的utils package中@incubator/nutils中，并且在以后对inc sdk的规划中，都需要做浏览器端和node端功能模块的分离，以免出现两端的兼容性问题

mongodb分片集群非片键唯一索引问题

分片集群由于其自身特点，对于非片键字段无法按照常规的方式创建unique索引。因为各分片相对独立，且可以维护自身分片的索引信息。针对整个集群来说，分片实例无法检测其他分片上的数据存储情况，导致unique检测失效。

解决方案：（proxy collection）

  • Unique Constraints on Arbitrary Fields
  • mongoose-unique-shard

npm install过程慢

安装过程中，有些包需要下载预编译好的二进制文件，这些npm包的部分功能由原生插件提供。预编译二进制包可能被托管在github.com上，国内网络访问可能会超时，导致npm install过程很慢，甚至失败。解决方案有：

  • 从源码编译：npm install –build-from-source。虽然不会从远程下载已经预编译好的包，但是从源码编译的过程也很慢，如果本地没有编译环境甚至可能会出错
  • 指定预编译包镜像源：每个包的镜像配置方式都不一样，部分包配置如下
  • 使用多阶段构建，让npm install单独作为一个构建阶段并生成一个镜像作为docker cache，如果package.json没有改动，将会直接命中缓存
  • 想过在docker中挂载宿主机上的npm cache目录，使在docker中执行npm install的时候能使用宿主机上的npm缓存，但是这种方法无效。因为docker build过程中无法进行volumn操作

正则表达式零宽断言的兼容性问题

正则表达式中的零宽断言中的负向零宽断言属于ES7的提案，并不是所有浏览器都支持，比如iOS端微信浏览器和小程序中就不支持。控制台将会报错“Invalid regular expression: invalid group specifier name”。解决方法是换一种正则表达式的写法，不使用零宽断言。

curl测试请求耗时

结果：0.244:1.044:2.672

时间指标解释 ：

time_connect——建立到服务器的 TCP 连接所用的时间

time_starttransfer——在发出请求之后，Web 服务器返回数据的第一个字节所用的时间

time_total——完成请求所用的时间

在发出请求之后，Web 服务器处理请求并开始发回数据所用的时间是——

（time_starttransfer）1.044 - （time_connect）0.244 = 0.8 秒

客户机从服务器下载数据所用的时间是——

（time_total）2.672 - （time_starttransfer）1.044 = 1.682 秒
