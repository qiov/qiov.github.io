背景

在项目迭代开发过程中，需求和缺陷涉及到的代码改动会比较多，特别在微服务架构模式下，涉及到的服务也会比较多。那么在版本发布过程中，可能会造成服务漏发或错发的情况。如果能够将每次的代码改动和tapd上的需求或缺陷单关联起来，在发版时，可以让开发同学作为参考，梳理出本次迭代需要发布的代码和服务。

注意：建议每次提交仅解决一个需求或者缺陷，不要在一次git提交中处理多个问题

commit msg规范说明：

  • 关联需求：--story=[story id1],[story id2] --user=[usernick] 描述信息
  • 关联缺陷：--bug=[bug id1],[bug id2] --user=[usernick] 描述信息
  • 无关联单据：--other=描述信息

修改git提交历史的commit msg

如果在push之前，已经提交了一些不符合规范的commit msg，可以通过以下方式来解决。

1、如果历史只有一条不符合规范的提交待push，可以执行git commit --amend进行修改

2、如果历史有超过一条不符合规范的提交待push，可以按以下流程修改提交msg

2.1、执行git rebase -i HEAD~3命令, 修改近三次的信息。这里的3视具体情况而定

2.2、找到不符合规范的日志, 将第一个pick 修改为edit, 然后 :wq 退出

2.3、执行git commit --amend, 修改第一条需要修改的msg。修改完成后, 执行git rebase --continue修改下一条不符合规范的日志

2.4、2.3执行完成之后，再查看日志记录，所有不符合规范的日志都被改正了，然后进行push操作
